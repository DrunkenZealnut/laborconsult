### Value Delivered

| 관점 | 내용 |
|------|------|
| Problem | 계산기 모듈이 실제 노동상담 사례의 답변과 얼마나 일치하는지 정량적 검증이 없음 |
| Solution | 114건 법원판례/상담사례에서 숫자 포함 질문을 추출 → 계산기에 입력 → 전문가 답변과 비교 |

| Core Value | 실제 노동상담 사례 기반 벤치마크로 계산기 신뢰성 입증 및 개선 방향 확보 |

## 1. 배경 및 목적

### 1.1 현황
- `output_legal_cases/` 디렉토리에 114건의 서울시 노동상담사례집 사례 존재
- 각 사례: 질문(구체적 근로조건) + 전문가 답변(법률 기반 계산 결과 포함)
- `wage_calculator/` 모듈에 23개 계산기 존재 (overtime, minimum_wage, severance 등)
- 이 둘을 비교한 정량적 벤치마크가 아직 없음

### 1.2 목적
1. 상담 사례의 질문에서 **숫자 데이터를 추출**하여 `WageInput` 구성
2. 해당 사례의 전문가 답변에서 **정답 금액 추출**
3. 계산기 모듈 결과와 정답 비교 → **일치율, 오차 범위** 측정
4. 계산기가 틀리는 패턴(유형/원인) 분류 → 개선 우선순위 도출

### 1.3 대상 데이터 분석

| 항목 | 수량 |
|------|------|
| 전체 사례 | 114건 |
| `## 질문` 섹션 보유 | 104건 |
| `## 답변` 섹션 보유 | 97건 |
| 계산기 관련 키워드 포함 (수당/임금/퇴직/보험 등) | 77건 |

**계산기 관련 사례 주제 분류 (예상)**:

| 주제 | 관련 계산기 | 예상 건수 |
|------|------------|----------|
| 연장/야간/휴일수당 | overtime | ~15건 |
| 최저임금 위반 여부 | minimum_wage | ~8건 |
| 주휴수당 | weekly_holiday | ~10건 |
| 연차수당 | annual_leave | ~12건 |
| 퇴직금 | severance | ~10건 |
| 통상임금/평균임금 | ordinary_wage, average_wage | ~8건 |

## 2. 기술 접근 방식

### 2.1 전체 파이프라인

```
[Phase 1] 사례 파싱 & 분류
  output_legal_cases/*.md → parse → {질문, 답변, 메타데이터}
  → Claude로 구조화: {임금형태, 임금액, 근로시간, 사업장규모, ...}

[Phase 2] WageInput 생성
  구조화된 데이터 → WageInput 변환
  → 계산 가능 여부 판별 (정보 충분/불충분)

[Phase 3] 계산기 실행 & 정답 추출
  WageInput → WageCalculator.calculate() → 계산기 결과
  답변 텍스트 → Claude로 정답 금액 추출

[Phase 4] 비교 & 리포트
  계산기 결과 vs 정답 → 일치/오차/불일치 분류
  → 벤치마크 리포트 생성
```

### 2.2 Claude를 활용한 데이터 추출

사례의 질문/답변은 자유형 한국어 텍스트이므로, Claude Haiku로 구조화 추출:

**질문 → 입력 데이터 추출 프롬프트**:
```
이 노동상담 질문에서 계산에 필요한 정보를 JSON으로 추출하세요:
- wage_type: 시급/일급/월급/연봉/포괄임금제
- wage_amount: 임금액 (숫자)
- daily_work_hours: 1일 근무시간
- weekly_work_days: 주 근무일수
- overtime_hours: 연장근로시간
- night_hours: 야간근로시간
- business_size: 사업장 규모
- service_years: 근속년수
- calculation_type: 연장수당/최저임금/주휴수당/퇴직금/...
```

**답변 → 정답 추출 프롬프트**:
```
이 전문가 답변에서 구체적인 금액/수치를 추출하세요:
- 계산 결과 금액들 (시급, 일급, 월급, 수당액 등)
- 계산 공식 (있는 경우)
- 판단 결과 (최저임금 충족 여부, 수급 가능 여부 등)
```

### 2.3 비교 기준

| 비교 항목 | 일치 기준 | 비고 |
|-----------|----------|------|
| 금액 비교 | 오차 ±5% 이내 = 일치 | 반올림/절사 차이 허용 |
| 시급 비교 | 오차 ±10원 이내 = 일치 | 소수점 처리 차이 |
| Yes/No 판단 | 정확 일치 | 최저임금 충족, 수급 가능 등 |
| 일수 비교 | 정확 일치 | 연차일수 등 |

### 2.4 결과 분류

| 분류 | 정의 |
|------|------|
| MATCH | 계산기 결과 = 정답 (허용 오차 내) |
| MISMATCH | 계산기 결과 ≠ 정답 (허용 오차 초과) |
| PARTIAL | 일부 항목 일치, 일부 불일치 |
| SKIP_INPUT | 질문에서 입력 데이터 추출 불가 |
| SKIP_ANSWER | 답변에 구체적 숫자 없음 (법률해석만 포함) |
| NOT_SUPPORTED | 해당 계산 유형이 계산기에 미구현 |

## 3. 구현 범위

### 3.1 구현할 것

1. **`benchmark_legal_cases.py`** — 메인 벤치마크 스크립트
   - 사례 파일 파싱 (질문/답변 분리)
   - Claude Haiku로 입력 데이터 & 정답 추출
   - WageInput 생성 & 계산기 실행
   - 결과 비교 & JSON 리포트 생성

2. **벤치마크 리포트** — `benchmark_results.json` + 콘솔 요약
   - 전체 일치율 (MATCH / total)
   - 계산기 유형별 일치율
   - MISMATCH 건의 상세 (사례번호, 기대값, 실제값, 오차)
   - 개선 우선순위 추천

### 3.2 구현하지 않을 것

- 계산기 모듈 자체의 수정 (벤치마크 결과 기반으로 별도 이슈화)
- 법률해석만 있는 사례의 평가 (숫자 없는 답변은 SKIP)
- 사례 데이터 크롤링 (이미 존재하는 114건 사용)
