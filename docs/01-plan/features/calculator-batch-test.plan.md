# Plan: 계산기 모듈 배치 테스트 (500건 추출 및 결과 보고)

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 기존 compare_calculator.py v2는 100건/8유형만 커버하며 79%가 비교불가 → 계산기 정확도 검증 불충분 |
| **Solution** | 500건 층화 샘플링 + 19개 계산기 유형 확장 + WageInput 추출 개선으로 포괄적 배치 테스트 실행 |
| **Function UX Effect** | 계산기별 정확도/오류율/비교불가율 정량화 → 우선 개선 대상 식별 가능 |
| **Core Value** | 실제 Q&A 데이터 기반 계산기 신뢰도 검증으로 챗봇 답변 품질 향상의 정량적 근거 확보 |

## 2. 목표

### 2.1 핵심 목표

1. **500건 층화 샘플링** — 계산 유형별 비례/최소 보장 샘플링
2. **19개 계산기 커버리지** — 기존 8개 → 전체 계산기 모듈 테스트
3. **비교불가율 50% 이하** — WageInput 추출 로직 개선으로 실질 비교 비율 향상
4. **정량적 보고서** — 계산기별 정확도/오차 분포/에러 패턴 분석

### 2.2 성공 기준

| 지표 | 목표 |
|------|------|
| 총 테스트 건수 | ≥ 500건 |
| 계산기 유형 커버리지 | ≥ 12개 유형 |
| 비교불가율 | ≤ 50% (현재 79%) |
| ✅+⚠️ (±20% 이내) 비율 | ≥ 60% (비교가능 건 기준) |
| 보고서 포함 항목 | 유형별 정확도, 에러 분류, 개선 우선순위 |

### 3.2 Phase 2: WageInput 추출 개선

**현재 `build_wage_input_from_markdown()` 한계:**
- 퇴직금: `start_date`, `end_date`, `average_wage` 추출 없음
- 실업급여: `age`, `insured_months`, `daily_wage` 추출 없음
- 육아휴직: `parental_leave_months`, `is_second_parent` 추출 없음

**개선 항목:**
1. 퇴직금용 근무기간/퇴직일 추출 (`_guess_start_date` 강화)
2. 실업급여용 연령/피보험기간/이직일 추출
3. 육아휴직용 월수/급여 추출
4. `provided_info` 필드 매핑 확장 (analysis_qna.jsonl 스키마 활용)

### 3.3 Phase 3: 비교 로직 확장

**기존 유형별 비교 로직 (compare_calculator.py):**
- ✅ 연장수당, 최저임금, 주휴수당, 연차수당, 해고예고수당, 통상임금, 임금계산, 일할계산

**신규 비교 로직:**
- 퇴직금: `result.summary["퇴직금"]` vs 답변 금액
- 실업급여: `result.summary["실업급여(일)"]` vs 답변 금액/일수
- 육아휴직: `result.summary["육아휴직급여(월)"]` vs 답변 금액

### 3.4 Phase 4: 보고서 생성

**출력 형식:**
1. **콘솔 보고서** — 전체 요약 + 유형별 통계 + 불일치 케이스 상세
2. **JSON 결과** — `batch_test_results.json` (500건 상세)
3. **분석 보고서** — `batch_test_report.md` (마크다운 보고서)

**보고서 포함 항목:**
- 전체 정확도 (✅/⚠️/❌/⏭️ 분포)
- 계산기별 정확도 히트맵
- 비교불가 원인 분류 (입력 추출 실패 / 답변 수치 부재 / 계산기 오류)
- 불일치 패턴 분석 (오차 분포, 반복 에러 유형)
- 개선 우선순위 매트릭스 (건수 × 오차율)

## 4. 기술 설계 개요

### 4.1 아키텍처

```
analysis_qna.jsonl ──→ 층화 샘플링 (500건)
                           │
                    ┌──────┴──────┐
                    ↓             ↓
              output_qna/     provided_info
              (마크다운)       (JSON 필드)
                    │             │
                    └──────┬──────┘
                           ↓
               build_wage_input_from_markdown()
                    (개선된 추출 로직)
                           │
                    ┌──────┴──────┐
                    ↓             ↓
              WageInput 생성   추출 실패 → ⏭️ 기록
                    │
                    ↓
           WageCalculator.calculate(inp, targets)
                    │
                    ┌──────┴──────┐
                    ↓             ↓
              계산 성공        계산 오류 → ⏭️ 기록
                    │
                    ↓
           답변 수치 추출 + 비교
                    │
                    ↓
           CompareResult (✅/⚠️/❌/⏭️)
                    │
                    ↓
           보고서 생성 (JSON + MD + 콘솔)
```

### 4.2 파일 구조

```
calculator_batch_test.py          # 메인 배치 테스트 스크립트 (신규)
batch_test_results.json           # 500건 상세 결과 (출력)
batch_test_report.md              # 분석 보고서 (출력)
compare_calculator.py             # 기존 v2 (수정하지 않음 — 참조용 유지)
```

## 5. 의존성 및 제약사항

### 5.1 의존성
- `analysis_qna.jsonl` — 이미 생성 완료 (9,699건)
- `output_qna/` — 9,809개 마크다운 파일 존재
- `wage_calculator/` — 26개 계산기 모듈 구현 완료

### 5.2 제약사항
- **답변 수치 추출 한계**: Q&A 답변이 정성적(법률 해석)인 경우 수치 비교 불가
- **입력 데이터 불완전성**: 질문에 필요 정보가 부족한 경우 WageInput 생성 불가
- **연도별 법령 차이**: 2010~2026년 게시글이 혼재 — `reference_year` 정확 매핑 필요
- **계산기 간 의존성**: ordinary_wage가 기반 → 통상임금 계산 오류 시 연쇄 영향

### 5.3 리스크

| 리스크 | 영향 | 완화 전략 |
|--------|------|-----------|
| 비교불가율이 여전히 높음 | 통계 유의미성 저하 | 추출 로직 단계적 개선, 비교불가 원인 분류 |
| 퇴직금/실업급여 추출 복잡 | 개발 시간 증가 | 기본 필드만 추출, 나머지는 ⏭️ 처리 |
| 계산기 오류 발견 시 | 수정 범위 확대 | 보고서에 기록만 하고 별도 PDCA로 수정 |
