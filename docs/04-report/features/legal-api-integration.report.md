## 3. Design 요약

### 3.1 핵심 설계

- **모듈 구조**: `legal_api.py` 단일 모듈에 API 클라이언트, XML 파싱, 캐시, 파싱 함수 통합
- **캐시 전략**: `_MST_CACHE` (영구/세션) + `_ARTICLE_CACHE` (24h TTL)
- **에러 처리 원칙**: 법령 API는 보조 소스 — 실패 시 항상 기존 RAG+LLM 방식으로 동작

## 6. 발견 사항 및 교훈

### 6.1 구현 중 발견된 문제 (6건)

| # | 문제 | 원인 | 해결 |
|---|------|------|------|
| 1 | API 인증 실패 | `OC` 파라미터에 API 키 해시가 아닌 **사용자 ID**를 넣어야 함 | `.env`에서 `LAW_API_KEY=kcsvictory`로 변경 |
| 2 | IP 미등록 | law.go.kr는 등록된 IP에서만 호출 허용 | 개발자 포털에서 `118.33.130.107` 등록 |
| 3 | MST 값 전부 오류 | 설계 시 추정값 사용 → 실제 API 응답과 불일치 (예: 270551=시행령, 265959=본법) | 17개 법령 모두 라이브 API로 검증 후 교체 |
| 4 | 조의N 조문 미반환 | XML `조문번호`가 "76"으로만 표시 → 제76조와 제76조의2 구분 불가 | `조문가지번호` XML 태그 활용 + `조문여부="전문"` 스킵 |
| 5 | 긴 법명 파싱 실패 | 정규식 `[\w·]+`가 공백 포함 법명 미매칭 | `[\w·ㆍ][\w·ㆍ\s]*?(?:법률|법|령|규칙)` 비탐욕 패턴 |
| 6 | middle dot 불일치 | `·` (U+00B7) vs `ㆍ` (U+318D) | API는 `ㆍ` 사용 → 약칭 매핑에서 `ㆍ` 통일 |

### 6.2 잘된 점

1. **Graceful degradation 설계**: 법령 API 장애 시에도 기존 RAG+LLM 정상 동작 — 사용자 영향 제로
2. **2단계 캐시**: L1(인메모리) + L2(Supabase)로 콜드스타트에도 빠른 응답
3. **Phase 2 조기 구현**: 판례·헌재 검색까지 한 번에 완성하여 추가 작업 불필요
4. **Circuit Breaker**: 연속 실패 시 30초 차단으로 불필요한 타임아웃 대기 방지
5. **0 iteration 완성**: 설계 단계의 철저함이 구현을 용이하게 함

### 6.3 재사용할 패턴

| 패턴 | 설명 | 적용 상황 |
|------|------|----------|
| 2단계 캐시 | L1 인메모리 + L2 영속(DB) | 정부/공공 API 연동 시 |
| Circuit Breaker | 연속 N회 실패 → 쿨다운 → half-open 재시도 | 외부 API 의존 모듈 |
| ThreadPoolExecutor 병렬 | 독립적 API 호출 N건 동시 실행 | 복수 자원 조회 시 |
| 약칭 정규화 | `_LAW_NAME_ALIASES` 딕셔너리로 입력 변형 통일 | 사용자 입력 다양성 대응 |
| MST 사전매핑 | 자주 조회하는 식별자를 하드코딩 | 외부 ID 기반 조회 최적화 |

### 6.4 주의사항

1. **외부 API 동적 데이터는 사전 검증 필수**: MST 값처럼 문서 기재값과 실제 값이 다를 수 있음
2. **API 문서와 실제 동작 차이**: IP 등록 요구사항, OC 파라미터 의미 등 문서 불충분
3. **XML 구조 파악**: 복잡한 XML은 샘플 데이터로 먼저 파싱 테스트 필요
4. **유니코드 변형 주의**: middle dot 같은 외형 동일·코드 상이 문자 확인
